### 12. 23+ spec drift resolution proposals

Это главный артефакт этого этапа. Для каждого SD-XXX item — propose resolution. Format:

| SD-XX | Description | Suggested resolution | Where | Effort |
|---|---|---|---|---|
| SD-9 | Backup contract drift (B2 declared, not implemented) | Update SESSION_CONTEXT line 746 to reflect real state until DB-003 fix lands; tag with [unimplemented] marker | SESSION_CONTEXT line 746 | 5 min |

| etc. |

Categories of resolution:
- Update doc (most common) — spec лгёт о реальности, doc fix
- Acknowledge limitation — spec was aspirational, mark [planned] / [partial] / [unimplemented]
- Fix code — реально кто-то forgot to update code per spec (rare — most drift is doc-lagging-behind)
- Delete obsolete — declared feature removed but doc still mentions
- Cross-reference fix — point to authoritative source (e.g. specific code file)

## Lessons learned
- Гибридная стратегия (12 audits + then fix) дала ~50+ cross-audit overlap pairs
- One-fix-many-wins backlog covers ~50% of findings одной серией PR
- 0 critical в 8 из 12 этапов — production posture overall solid
- Critical findings всегда были в одних областях: backup integrity, SPA-trap protection — нужна defensive infrastructure
- Verified safe сections (300+ items суммарно) — solid foundation для next-year аудита

## Severity для documentation

- critical = docs misleading operator на production-critical task (DB-001-003 SESSION_CONTEXT B2 backup лжёт). Spec drift items pointing to это count.
- high = important section missing (restore drill, secret rotation SOP), 23+ unresolved spec drift items overall, ai-context/* размер >> recommended bench (12K tokens)
- medium = inconsistent wording между files, missing cross-links, outdated examples
- low = formatting nits, typos
- info = future doc improvements (architecture diagrams, API docs, contributor guide)

## Правила работы

- Ничего не правь — не код, не documentation. Только findings + propose resolutions.
- Не комить, не пушить, не деплоить.
- Spec drift resolutions — это proposals, не applied changes. Оператор сам выберет какой path для каждого SD item.
- Integrated backlog в INDEX.md — это рекомендация, оператор может перешерстить приоритеты.
- Assumption'ы — ⚠ assumes:
- В конце сессии обнови WORKLOG.md финальным entry за всю серию (12 этапов, ~150-200 findings, 19 bundle targets, summary verdict).
